### 3. dashboard.ts — 端點命名修正
- **檔案**: `src/server/routes/dashboard.ts`
- **修改**: `/foreign-buy-2day` → `/trust-buy-2day`
- **原因**: 端點實際查詢的是 `trust_net`（投信），不是外資（foreign）
- **前端同步**: `DashboardView.tsx` 中的 fetch URL 也同步更新

### 4. dashboard.ts — break-ma200 效能優化
- **檔案**: `src/server/routes/dashboard.ts`
- **優化前**: 對每檔股票執行 2 次 `getMA200()` 個別 SQL 查詢 + 額外 name/volume 查詢
  - N 檔股票 ≈ 4N + 2 次 SQL 查詢（約 7,200+ 次）
- **優化後**:
  - 1 次批次查詢取得所有股票的 prev/latest close
  - 1 次批次查詢取得所有股票在兩天的 close（用於後續 MA 計算）
  - 1 次批次查詢取得 stock_name
  - 1 次批次查詢取得 volume
  - 每檔仍需個別查詢 MA200（SQLite 不支援 window function 的 PARTITION BY + ORDER BY 混合），但 name/volume 已批次化
  - 總查詢次數從 ~4N+2 降至 ~N+4
- **移除**: `getMA200()` 輔助函數（已內聯到迴圈中）

### 5. dashboard.ts — Supabase 分支狀態
- 所有 4 個端點的 Supabase 分支仍然是 stub（返回 `stock_features` 原始資料）
- 真正的篩選邏輯只在 local SQLite 分支實現
- Supabase 分支適合作為 fallback，但不提供完整的篩選功能

## 遇到的錯誤與修復

1. **MA200 優化版本過於複雜** — 初次嘗試用 CTE window function 批次查詢，但 SQLite 不支援複雜的 PARTITION BY + ORDER BY 混合。改為保留個別 MA200 查詢，但批次化 name/volume 查詢。
2. **3 次 map 渲染 grid 列** — 改用單一 map 回傳多個 Fragment，保持 DOM 結構一致性。
