# WORKFLOW.md - 线性工作流

本文件是流程说明，不默认读取。正常情况下只看 `AGENTS.md` 即可。

## 第 1 步：PRD 规划

目标：把用户想法压缩成一个清楚、可交付、可验收的 P0。

输入：

- 用户原始想法。
- 必要时，用户补充的场景、目标用户、约束。

输出：

- `PRD.md`

完成条件：

- 一句话定位清楚。
- 核心用户和场景清楚。
- P0 / P1 / 不做什么清楚。
- 验收标准可观察。
- 用户明确确认 PRD。

## 第 2 步：项目框架

目标：决定项目怎么组织，而不是马上写代码。

输入：

- `PRD.md`

输出：

- `PROJECT_FRAME.md`
- `GATES.md` 草案

完成条件：

- 技术栈和理由清楚。
- 目录结构清楚。
- 模块边界清楚。
- 数据模型或数据流清楚。
- 实现切片清楚。
- 用户确认不可逆技术决定。

**后端 / 全栈项目的额外要求：**

如果项目需要后端，`PROJECT_FRAME.md` 第 6 节（API/Mock 判断）必须在进入第 3 步前单独确认：

- 是否需要真实 API，还是前端先用 mock。
- API 契约写在哪里（PROJECT_FRAME.md 内联 / 单独 CONTRACT.md）。
- 前后端的实现顺序（并行 / 前端先行 / 后端先行）。

这些决策会影响第 3 步交互逻辑的细节，必须提前锁定。

**约束冲突处理：**

如果本步骤发现技术约束与 PRD 的产品范围冲突（例如某功能在选定技术栈下不可行），立即退回第 1 步，标注冲突点，等用户确认范围或技术栈其中一个改变后再推进。

## 第 3 步：交互逻辑

目标：定义用户怎么操作、系统怎么反馈、状态怎么流转。

输入：

- `PRD.md`
- `PROJECT_FRAME.md`

输出：

- `FLOWS.md`

完成条件：

- 页面或入口清楚。
- 每个关键动作的触发、反馈、结果清楚。
- 空、加载、错误、成功状态清楚。
- 异常路径清楚。
- 用户确认交互。

**约束冲突处理：**

如果本步骤发现交互设计与技术边界冲突，退回第 2 步，而不是在第 3 步自行决定技术取舍。

## 第 4 步：前端交接包

目标：生成一份可以直接发给专业前端 AI 的自包含文档。

输入：

- `PRD.md`
- `PROJECT_FRAME.md`
- `FLOWS.md`
- `GATES.md`

输出：

- `FRONTEND_HANDOFF.md`

完成条件：

- 前端 AI 不读其他文档也能理解目标、页面、交互、数据、限制和验收标准。
- 不含含糊口头约定。
- 用户确认可以交给前端 AI。

## 第 5 步：专业前端 AI 生成代码

目标：让前端实现尽量只依赖 `FRONTEND_HANDOFF.md`。

输入：

- `FRONTEND_HANDOFF.md`
- 相关源码
- 必要时 `check.sh`

输出：

- 前端代码
- 必要的 mock 数据

完成条件：

- 界面行为符合交接包。
- 不擅自改需求、交互和技术边界。
- 可以运行基础检查。

## 第 6 步：项目记录

目标：让下一次对话快速接续，不靠聊天记录。

输入：

- 本轮完成内容。
- 报错或检查结果。
- 当前版本变化。

输出：

- `STATUS.md`
- `CHANGELOG.md`
- `ERRORS.md`

完成条件：

- 当前版本和下一步清楚。
- 已完成内容有版本记录。
- 报错有复现、根因、处理、预防规则。

## 第 7 步：验收闸门

目标：用脚本和验收标准判断是否完成。

输入：

- `GATES.md`
- `check.sh`
- 必要时 `PRD.md` 验收标准

输出：

- 验收结果
- 需要修复的问题

完成条件：

- `bash check.sh` 通过。
- P0 验收标准满足。
- 如有 UI，交互符合 `FLOWS.md` / `FRONTEND_HANDOFF.md`。
