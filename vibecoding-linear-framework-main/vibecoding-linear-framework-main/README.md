# VibeCoding 线性框架

这是一套低 token、强执行的 AI 开发流程框架。

它的核心不是让 AI 每次读取所有文档，而是让 AI 按固定流程推进，并在每一步只读取必要文件。

## 流程

1. PRD 规划
2. 项目框架
3. 交互逻辑
4. 前端交接包
5. 专业前端 AI 生成代码
6. 项目记录
7. 验收闸门

## 文件说明

| 文件 | 作用 | 何时读取 |
|---|---|---|
| `AGENTS.md` | 唯一默认入口，负责状态和文件路由 | 每次默认读取 |
| `WORKFLOW.md` | 详细流程说明 | 需要理解流程时 |
| `PRD.md` | 产品定义 | PRD 规划、验收口径不清楚时 |
| `PROJECT_FRAME.md` | 项目框架和技术方案 | 定技术方案、目录、模块时 |
| `FLOWS.md` | 交互逻辑 | 定页面、流程、状态时 |
| `FRONTEND_HANDOFF.md` | 发给专业前端 AI 的自包含交接包 | 前端实现前 |
| `STATUS.md` | 当前版本和当前状态 | 接续工作、记录进度时 |
| `CHANGELOG.md` | 历史版本和版本号 | 查历史、发版时 |
| `ERRORS.md` | 历史报错和处理记录 | 修 bug、复盘时 |
| `GATES.md` + `check.sh` | 验收标准和可执行检查 | 开发和验收时 |

## 使用原则

- 默认只读 `AGENTS.md`。
- 每一步只读路由表要求的文件。
- PRD、项目框架、交互逻辑都确认后，再生成前端交接包。
- 前端交接包必须自包含，专业前端 AI 只读它也能开工。
- 当前状态、历史版本、历史报错分开记录，避免一个文件变成 token 黑洞。
- 完成以 `bash check.sh` 和验收标准为准。

## 全栈 / 后端项目

本框架默认流程是前端友好的，但支持全栈项目。核心差异：

- 第 2 步（项目框架）完成后，需单独确认 `PROJECT_FRAME.md` 第 6 节的 API/Mock 决策，再进入第 3 步。
- 前后端实现顺序（并行 / 前端先行 / 后端先行）必须在第 2 步明确。
- 如果需要单独管理 API 契约，可在 `PROJECT_FRAME.md` 旁边创建 `CONTRACT.md`，并更新路由表。

## 约束冲突与回退

发现约束冲突时，不绕过——退回冲突涉及的最早步骤：

```
交互超出技术边界  →  退回第 2 步（项目框架）
技术约束影响产品范围  →  退回第 1 步（PRD）
```

冲突必须明确标注后由用户确认，才能继续推进。

## 接续已有项目

新对话开始时：

1. 读 `AGENTS.md` 状态块，确认当前步骤。
2. 如果 `ERRORS.md` 有记录，一并读取（避免重复踩坑）。
3. 根据路由表读取当前步骤所需文件，开始工作。
